# ADR-001: Adopt ELT Architecture using PostgreSQL and dbt

**Date:** July 2026  
**Status:** Accepted  
**Deciders:** PipeOne Development Team

---

## Context

The GitHub Events API returns deeply nested JSON with variable structure across event types. A typical event contains 10+ levels of nesting: event metadata, payload structure (which differs by event type), and nested user/repository objects.

Three architectural patterns were possible:

1. **Transform in Python:** Parse JSON immediately during ingestion, validate, and load normalized data
2. **ETL Pipeline:** Transform before loading (external process), then load clean data
3. **ELT Pipeline:** Load raw data as-is, transform afterward using SQL

### Why Raw Data Matters

During analytics development, business requirements change. Questions like "What fields do we actually need?" are answered iteratively, not upfront. If we transform during ingestion:
- Re-running the pipeline requires re-ingesting from the API
- Changes to transformation logic require re-processing everything
- Debugging is difficult—we've already discarded the original structure

By preserving raw JSON in PostgreSQL, we:
- Avoid re-ingestion when requirements change
- Keep the original data for forensics and debugging
- Re-run transformations instantly (dbt is fast)
- Explore the data safely without losing information

### Why ELT Fits Analytics

Analytics pipelines prioritize flexibility and iteration. The transformation layer evolves as analysts ask new questions. ETL systems are rigid—they assume the transformation is stable. ELT systems are flexible—transformation is SQL in version control, re-runnable on demand.

---

## Decision

We adopt an **ELT architecture** using:
- **Extract:** Python `github_client.py` fetches events from the GitHub Events API
- **Load:** Raw JSON stored immediately in PostgreSQL as JSONB in the `github_events_raw` table
- **Transform:** dbt models transform data through Medallion layers (Bronze → Silver → Gold)

### Architecture Layers

```
GitHub API → Python Ingestion → PostgreSQL Raw (JSONB)
                                       ↓
                                  Bronze Layer
                              (stg_github_events)
                                       ↓
                                  Silver Layer
                        (int_push_events, int_pull_requests)
                                       ↓
                                  Gold Layer
                              (Future: aggregations)
```

### Key Technologies

**PostgreSQL JSONB:** Native JSON type that PostgreSQL can index and query efficiently. Operators like `->` (JSONB) and `->>` (TEXT) allow fine-grained field extraction. This enables exploration without schema changes.

**dbt:** Transforms raw data using SQL. Provides:
- `source()` macro: Documents data sources, auto-generates freshness tests
- `ref()` macro: Creates dependency tracking between models, enables lineage visualization
- `{{ ref('upstream_model') }}` ensures proper build order
- Automated testing: Schema tests (unique, not_null, accepted_values) and custom SQL tests

**Medallion Architecture:** Three-layer structure providing progressive refinement:
1. **Bronze:** Raw, unmodified data from the source
2. **Silver:** Cleaned, deduplicated, business-logic-free models
3. **Gold:** Aggregated, business-ready tables (future)

Each layer is independently queryable, enabling different use cases.

---

## Consequences

### Positive Outcomes

**Modular Pipeline**  
Each layer has a single responsibility. Bronze ingests and types, Silver flattens and standardizes, Gold aggregates. Changes to one layer don't break others.

**Easier Debugging**  
A bad metric in a dashboard? Query the Silver layer to see transformed data. See something wrong there? Query the Bronze layer to see the original JSON. Preserve the entire chain.

**Re-runnable Transformations**  
Running `dbt run` is instant—no API calls, no re-ingestion. We can test new transformations, refactor existing ones, and re-run them safely.

**Better Maintainability**  
Transformation logic is in SQL files in version control. Code reviews track who changed what. SQL is the analytics standard—easy for other analysts to understand and contribute.

**Separation of Concerns**  
Python handles API authentication and retry logic (stateful, complex). SQL handles data transformation (declarative, testable). Each tool does what it's best at.

**Built-in Testing**  
dbt runs 26 automated tests on every build: schema tests catch structural issues, custom tests catch business logic violations. Tests run before deployment—no bad data reaches dashboards.

### Trade-offs

**More SQL Models**  
With ELT, we write more SQL because transformation is explicit. Traditional ETL systems hide transformation logic in code. We trade complexity for visibility.

**Slightly More Project Complexity**  
Three layers (Bronze, Silver, Gold) require coordination. However, this structure also prevents complexity from concentrating in one place.

**JSONB Requires Understanding**  
Not all analysts are familiar with JSONB operators. Training is needed. However, JSONB knowledge is increasingly valuable across analytics.

## Alternatives Considered

### Alternative 1: Transform Everything in Python

Parse JSON in `github_client.py`, validate, and load normalized data immediately.

**Why not:** Re-running transformations requires re-ingesting from the API. Python is stateful and complex for data transformation logic. Changes to transformation require code review, testing, and deployment. Harder to iterate during development.

### Alternative 2: One Large SQL Query

Skip Bronze and Silver layers. Load raw JSON and transform directly into Gold tables in a single query.

**Why not:** Single transformation is monolithic and difficult to test. Debugging is hard—can't query intermediate stages. Changing a Gold model requires re-running the entire query. Violates separation of concerns. Medallion architecture exists precisely to break this into manageable pieces.

### Alternative 3: Traditional ETL (Transform Before Load)

Use Python to parse and normalize during ingestion, then load clean data.

**Why not:** Loses original data structure. Transformation is baked into ingestion logic—changing it requires re-running against the API. Inflexible for iterative analytics development. Easier to build, harder to maintain and extend.

---

## Summary

ELT with PostgreSQL and dbt provides the flexibility needed for analytics. Raw data in JSONB enables exploration without schema constraints. dbt's SQL-based transformations stay in version control and re-run instantly. The Medallion Architecture enforces clarity through progressive refinement. Automated testing ensures data quality. This foundation supports analytical growth as requirements evolve.
